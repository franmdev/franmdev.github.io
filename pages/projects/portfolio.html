Arquitectura del Sistema

4 componentes · responsabilidad única · sin acoplamiento entre capas

Cada capa puede desplegarse, testearse y escalarse de forma independiente. El frontend no conoce la lógica de seguridad; el backend no conoce el layout; la base de datos no conoce el endpoint.

  • FE GitHub Pages — CDN Global, CSS puro: HTML/CSS/JS nativo sin frameworks. CSS puro con arquitectura ITCSS en archivos separados por capa: 14 KB bundle, 0 compiladores, 0 build step. GitHub Pages entrega el sitio desde el CDN global de forma gratuita. Componentes (navbar/footer) cargados vía fetch() — un único archivo por componente, sin duplicar HTML. Dark mode con IIFE síncrona en <head> para eliminar flash de color.
  • BE Azure Functions v4 — Serverless Python 3.11: Endpoint /api/register_visitor orquesta el pipeline de seguridad completo. 9 casos de respuesta documentados. Deploy automático desde GitHub Actions en cada push a main. Connection pool persistente a PostgreSQL con SSL. Consumption Plan: $0 hasta el millón de ejecuciones/mes.
  • DB PostgreSQL 15 — Analytics en Tiempo Real: Dos tablas con roles separados: ip_lookup_cache como blocklist permanente y Fast Pass TTL 24h; visitors como histórico analítico con FK ON DELETE SET NULL. UPSERT idempotente en cada visita. Índices estratégicos en timestamp, país y IP. Limpieza automática diaria vía Timer Trigger. Tier B1ms: incluido en créditos Azure for Students.
  • SEC Pipeline de Seguridad — 6 Capas por Costo Creciente: La capa L0 (localStorage, 0ms, sin red) va primero y elimina el 90% del tráfico recurrente. La capa L4 (ipapi.is, ~$0.002/llamada) solo se invoca para IPs completamente desconocidas. El costo operativo de ~$5/mes resulta de este ordenamiento — no de limitar funcionalidad.

Pipeline de Seguridad — 6 Capas

Ordenadas por costo creciente · 9 casos documentados · fail-safe ante error de API

El principio de diseño: bloquear el tráfico inválido en la capa más barata posible. El costo variable (~$5/mes) es consecuencia directa de que el 90% del tráfico recurrente se resuelve en L0 (cliente, 0ms) antes de llegar al backend.

  • L0 Session Cache — localStorage, 0ms, cero red: El cliente verifica sessionStorage antes de cualquier request. TTL 30 min cubre la sesión de navegación activa. ~60% del tráfico resuelto sin tocar el backend.
  • L1 Cloudflare WAF — Geobloqueo en el edge: Regla de geolocalización que restringe a Chile. El bloqueo ocurre en el CDN de Cloudflare, sin cómputo Azure. Tráfico internacional descartado antes de llegar a la función.
  • L2 Rate Limiting — 15 req/min por IP: En memoria de la Azure Function. Protege contra enumeración automatizada. Trade-off conocido: se pierde al reciclar la instancia (~1×/día), aceptable para el perfil de tráfico de un portafolio personal.
  • L3 Cloudflare Turnstile — Challenge anti-bot silencioso: El usuario ve un spinner, no un CAPTCHA con imágenes. Turnstile evalúa señales del navegador en segundo plano. Solo muestra interacción visible si la puntuación de riesgo lo requiere.
  • L4 ipapi.is — Detección VPN/Proxy/Tor (6 flags): La capa con costo variable (~$0.002/req). Solo se invoca para IPs que pasaron L0–L3 y no están en caché de PostgreSQL. Evalúa: is_vpn, is_proxy, is_tor, is_datacenter, is_abuser, is_bot. Un flag positivo bloquea y registra en blocklist permanente.
  • L5 PostgreSQL Cache TTL 24h — Fast Pass: IPs validadas se guardan en ip_lookup_cache. En la siguiente visita: respuesta en ~50ms sin llamar a ipapi.is. IPs bloqueadas se conservan en blocklist permanente sin expiración. La base de datos es el estado persistente del sistema de seguridad.

Decisiones de Ingeniería

Alternativas evaluadas · impacto cuantificado · documentadas en el repositorio

  • D1 Caché en dos capas: localStorage 30min + PostgreSQL 24h
    Sin caché: ~$200/mes proyectados. Con caché cliente + servidor: ~$5/mes. Los TTLs son complementarios — 30 min cubre la sesión activa, 24h cubre la reincidencia entre sesiones. Redis descartado: costo adicional de infraestructura innecesario para un sistema de función única.
  • D2 CSS puro con arquitectura ITCSS — GitHub Pages gratuito, 0 build step
    CSS puro permite control total del estilo sin dependencias runtime, compatibilidad completa con GitHub Pages (hosting estático gratuito) y navegación instantánea para los visitantes. ITCSS organiza el CSS en capas de especificidad creciente para eliminar conflictos de cascada. SSGs (Eleventy, Hugo) descartados: overhead de build para 6 páginas.
  • D3 Dark mode con IIFE síncrona en <head>
    La IIFE aplica html.dark antes del primer frame, eliminando el FOUC. Fuente de verdad única: localStorage.getItem('theme') || 'dark'. @media prefers-color-scheme descartado: coordinar estado CSS con estado JS genera condiciones de carrera.
  • D4 Componentes dinámicos vía fetch() — patrón dataset.bound
    Navbar y footer se inyectan desde un único archivo HTML mediante load-components.js. El evento componentLoaded notifica a los demás módulos JS cuando el DOM está listo. El atributo data-bound en el nodo previene doble-binding de listeners cuando la inicialización puede ocurrir en dos momentos del ciclo de vida.
