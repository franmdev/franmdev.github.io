El Problema

El mercado B2B de recursos humanos genera miles de publicaciones diarias en plataformas de empleo, pero la gran mayoría no son ofertas laborales reales: son ruido — avisos de proveedores, contenido duplicado, publicaciones en otros idiomas o datos mal formateados por SPAs que truncan el DOM antes del render completo.

Construir un sistema capaz de extraer, limpiar, deduplicar y clasificar ese volumen de datos de forma confiable —sin infraestructura cloud cara y con resiliencia ante fallos de red— requería más que un script de scraping. Requería una arquitectura.

El objetivo: un pipeline capaz de operar de forma autónoma semanas seguidas, reportar su propio estado, y alertar al operador solo cuando realmente importa.

Arquitectura por Fases

FASE 1 Command & Control — ChatOps

Orquestación · Telemetría · Seguridad Operacional

El cerebro del sistema: un Telegram Daemon con RBAC por TELEGRAM_ADMIN_IDS que actúa como panel de control remoto desde cualquier dispositivo. Implementa apagado graceful mediante flags IPC (stop_signal.flag) y os.fsync() para bypass del caché SMB/Windows. Telemetría en tiempo real desde status.json con FileLock para acceso atómico. Screenshots forenses completos del escritorio con PyAutoGUI.

Python Telegram Bot API PyAutoGUI filelock os.fsync() python-dotenv RBAC

FASE 2 Data Collection — Extracción Biomimética

Scraping · Anti-Bot · Session Management

Motor de scraping sobre Selenium + Microsoft Edge con perfiles persistentes del SO. El enfoque anti-bot es arquitectónico: hereda cookies de sesión del perfil (no inyecta credenciales en runtime que detectaría el WAF). Implementa distribución Normal gaussiana para delays humanos eludiendo detección heurística. Self-healing ante Login Wall: detecta redirección, pausa táctica, alerta HITL vía Telegram y auto-termina si persiste. Workers paralelos por perfil de navegador.

Selenium Microsoft Edge Distribución Gaussiana Session Hijacking Prevention HITL Alerts subprocess

FASE 3 Bronze Layer — Ingesta Event-Driven

Data Lake · Spooler Asíncrono · Merge Histórico

Data Lake local en CSV con patrón Spooler asíncrono. El collector agrega datos en staging y el processor los consolida al volumen base. Incluye merge de datasets históricos y limpieza NLP agresiva: eliminación de emojis, URLs, HTML y truncamientos de SPA. Deduplicación exacta por hash antes de persistir.

pandas xxhash CSV Data Lake Event-Driven NLP Cleaning Merge Histórico

FASE 4 High Availability & Zero-Trust Sync

Proxmox · LXC Air-Gapped · rclone Cloud Sync

La fase más crítica de infraestructura. Resuelve el problema real de que el kernel Linux bloquea procesos en Estado D (Uninterruptible Sleep) cuando el punto de montaje SMB/CIFS desaparece abruptamente, paralizando el hipervisor Proxmox completo. Solución en dos capas: Capa 1 (elegante) — flag IPC + watchdog en LXC se apaga voluntariamente. Capa 2 (desfibrilador) — hook Perl en Proxmox con umount -f -l + pct stop asíncrono ante BSOD o corte eléctrico. rclone sincroniza a OneDrive/S3 con validación MD5 y backup incremental.

Proxmox VE LXC Containers rclone SMB/CIFS Perl Hooks systemd OneDrive / S3 MD5 Validation

FASE 5 Silver Layer — MDM & Data Engineering

Deduplicación Semántica · Filtrado Idioma · Pipeline 5 Etapas

Pipeline de deduplicación multicapa con dos algoritmos benchmarkeados: TF-IDF + Coseno CUDA (O(n²) distribuido en GPU, mayor precisión semántica) vs. MinHash + LSH (O(n), escalable, índice persistente en disco). Pipeline de 5 etapas: ① deduplicación exacta por hash, ② filtrado heurístico por criterios B2B, ③ near-duplicates semánticos, ④ verificación Levenshtein con difflib, ⑤ exportación Silver + reporte de auditoría. Filtro de idioma con voting ensemble de 5 capas (Lingua + langdetect + heurísticas).

TF-IDF CUDA MinHash LSH datasketch difflib Lingua langdetect xxhash scikit-learn

FASE 6 Gold Layer — MLOps & NLP

BETO Fine-tuning · Active Learning · WeightedTrainer

Fine-tuning de BETO (dccuchile/bert-base-spanish-wwm-cased) en modo air-gapped. WeightedTrainer con class_weights automáticos para dataset desbalanceado. FP16 mixed precision + padding dinámico para optimizar VRAM. Active Learning loop de 5 buckets por incertidumbre del modelo (<0.15 basura, 0.40–0.60 zona crítica QA, >0.85 oferta confirmada). Salida en Excel con formato condicional dinámico (openpyxl) para revisión HITL visual. Dataset V2 construido integrando ground truth V1 + batches revisados por Active Learning.

HuggingFace Transformers BETO Spanish BERT PyTorch CUDA / FP16 Active Learning WeightedTrainer openpyxl HITL
