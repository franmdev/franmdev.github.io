El Problema

El mercado B2B de recursos humanos genera miles de publicaciones diarias en plataformas de empleo, pero la gran mayoría no son ofertas laborales reales: son ruido — avisos de proveedores, contenido duplicado, publicaciones en otros idiomas o datos mal formateados por SPAs que truncan el DOM antes del render completo.

Construir un sistema capaz de extraer, limpiar, deduplicar y clasificar ese volumen de datos de forma confiable —sin infraestructura cloud cara y con resiliencia ante fallos de red— requería más que un script de scraping. Requería una arquitectura.

El objetivo: un pipeline capaz de operar de forma autónoma semanas seguidas, reportar su propio estado, y alertar al operador solo cuando realmente importa.

Arquitectura por Fases

FASE 1 Command & Control — ChatOps

Orquestación · Telemetría · Seguridad Operacional

El cerebro del sistema: un Telegram Daemon con RBAC por TELEGRAM_ADMIN_IDS que actúa como panel de control remoto desde cualquier dispositivo. Implementa apagado graceful mediante flags IPC (stop_signal.flag) y os.fsync() para bypass del caché SMB/Windows. Telemetría en tiempo real desde status.json con FileLock para acceso atómico. Screenshots forenses completos del escritorio con PyAutoGUI.

FASE 2 Data Collection — Extracción Biomimética

Scraping · Anti-Bot · Session Management

Motor de scraping sobre Selenium + Microsoft Edge con perfiles persistentes del SO. El enfoque anti-bot es arquitectónico: hereda cookies de sesión del perfil en lugar de inyectar credenciales en runtime, lo que detectaría el WAF. Implementa distribución Normal gaussiana para delays humanos eludiendo detección heurística. Self-healing ante Login Wall: detecta redirección, pausa táctica, alerta HITL vía Telegram y auto-termina si persiste. Workers paralelos por perfil de navegador.

FASE 3 Bronze Layer — Ingesta Event-Driven

Data Lake · Spooler Asíncrono · Merge Histórico

Data Lake local en CSV con patrón Spooler asíncrono: el collector agrega datos en staging y el processor los consolida al volumen base. Incluye merge de datasets históricos y limpieza NLP agresiva: eliminación de emojis, URLs, HTML y truncamientos de SPA. Deduplicación exacta por hash antes de persistir.

FASE 4 High Availability & Zero-Trust Sync

Proxmox · LXC Air-Gapped · rclone Cloud Sync

La fase más crítica de infraestructura. Resuelve el problema real del kernel Linux bloqueando procesos en Estado D (Uninterruptible Sleep) cuando el punto de montaje SMB/CIFS desaparece abruptamente, paralizando el hipervisor Proxmox completo. Solución en dos capas: Capa 1 (elegante) — flag IPC + watchdog en LXC se apaga voluntariamente antes de cualquier I/O peligroso. Capa 2 (desfibrilador) — hook Perl en Proxmox con umount -f -l + pct stop asíncrono ante BSOD o corte eléctrico. rclone sincroniza a OneDrive/S3 con validación MD5 y backup incremental.

Proxmox VE LXC Containers rclone SMB/CIFS Perl Hooks systemd OneDrive / S3 MD5 Validation

FASE 5 Silver Layer — MDM & Data Engineering

Deduplicación Semántica · Filtrado Idioma · Pipeline 5 Etapas

Pipeline de deduplicación multicapa con dos algoritmos benchmarkeados: TF-IDF + Coseno CUDA (O(n²) distribuido en GPU, mayor precisión semántica) vs. MinHash + LSH (O(n), escalable, índice persistente en disco). Pipeline de 5 etapas: ① deduplicación exacta por hash, ② filtrado heurístico por criterios B2B, ③ near-duplicates semánticos, ④ verificación Levenshtein con difflib, ⑤ exportación Silver + reporte de auditoría. Filtro de idioma con voting ensemble de 5 detectores (Lingua + langdetect + heurísticas).

TF-IDF CUDA MinHash LSH datasketch difflib Lingua langdetect xxhash scikit-learn

FASE 6 Gold Layer — MLOps & NLP

BETO Fine-tuning · Active Learning · WeightedTrainer

Fine-tuning de BETO (dccuchile/bert-base-spanish-wwm-cased) en modo air-gapped. WeightedTrainer con class_weights automáticos para dataset desbalanceado. FP16 mixed precision + padding dinámico para optimizar VRAM. Active Learning loop de 5 buckets por incertidumbre del modelo (<0.15 descartado, 0.40–0.60 zona crítica QA, >0.85 oferta confirmada). Salida en Excel con formato condicional dinámico (openpyxl) para revisión HITL visual. Dataset V2 construido integrando ground truth V1 + batches revisados por Active Learning.

Decisiones de Ingeniería de Datos

Cada decisión fue evaluada por impacto medible en producción o benchmarkeada localmente. Estas cuatro son las más representativas del enfoque de diseño.

xxhash vs hashlib para deduplicación exacta

Rendimiento · Throughput · Fases 3 & 5

Alternativa evaluada: hashlib (SHA-256 / MD5) — estándar de librería, sin dependencias adicionales, criptográficamente seguro.
Decisión: xxhash. 3× más rápido en benchmarks locales para strings de longitud variable (títulos + descripciones de empleo). No criptográfico, pero suficiente para deduplicación por contenido donde el adversario no existe. El ahorro de ciclos acumulado en millones de registros es significativo en CPU sin GPU dedicada para esta tarea.

xxhash hashlib Benchmark local Throughput

TF-IDF CUDA vs MinHash/LSH para near-duplicates

Precisión vs Escalabilidad · O(n²) vs O(n) · Fase 5

Problema: near-duplicates semánticos no detectables por hash exacto — el mismo anuncio con variaciones menores de texto.
TF-IDF + Coseno CUDA: O(n²) sobre GPU, alta precisión semántica, índice no persistente — requiere GPU disponible en cada ejecución.
MinHash + LSH: O(n) aproximado, índice persistente en disco, escalable sin GPU, tolerancia configurable por número de permutaciones.
Decisión final: ambos disponibles como módulos intercambiables. TF-IDF para runs de limpieza profunda con GPU disponible; MinHash/LSH para ejecución continua sin GPU.

TF-IDF Cosine Similarity MinHash LSH datasketch scikit-learn CUDA

IPC Flag vs SIGTERM para apagado graceful

Resiliencia · Estado D · Fases 1 & 4

Problema: el kernel Linux coloca procesos en Estado D (Uninterruptible Sleep) cuando el punto de montaje SMB/CIFS desaparece. SIGTERM no interrumpe Estado D — el proceso queda zombi y bloquea el hipervisor completo.
Decisión: señalización mediante flag de archivo (stop_signal.flag) con os.fsync() para garantizar escritura inmediata en filesystems de red. El daemon Python lee el flag en cada iteración y ejecuta cleanup antes de cualquier operación I/O peligrosa. SIGTERM queda como fallback únicamente en el hook Perl del hipervisor ante fallos catastróficos.

IPC Flag os.fsync() Estado D Linux SMB/CIFS Graceful Shutdown Proxmox VE

Active Learning vs etiquetado completo del dataset

MLOps · Calidad del dato · Dataset Versioning · Fase 6

Problema: etiquetar manualmente miles de publicaciones es inviable. Un clasificador entrenado sobre datos desbalanceados sin estrategia produce alta precisión en la clase mayoritaria y bajo recall en la que importa (ofertas reales).
Decisión: Active Learning loop de 5 buckets por probabilidad de incertidumbre. Solo se revisan manualmente los registros en la zona crítica (0.40–0.60); los extremos se etiquetan automáticamente. WeightedTrainer corrige el desbalance con class_weights calculados desde la distribución real del dataset. Dataset V2 = V1 + batches revisados, con trazabilidad completa de origen por registro.
